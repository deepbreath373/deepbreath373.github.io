缓存穿透、缓存击穿、缓存雪崩

发表于 2021-11-02 更新于 2022-01-08 分类于 Redis
本文字数： 1.2k

有关 Redis 缓存存在的问题

缓存穿透

描述：

缓存穿透是指缓存和数据库中都没有要查找的数据，而用户不断地发起请求就导致在缓存和数据库上进行了无用的查询，导致数据库的压力增加。

解决方案：

  1. 可以在用户发起请求时对请求进行校验，如果不合法，可以直接拒绝请求，避免本次请求落在数据库上。
  2. 从缓存和数据库中都没有读取到的数据，可以暂时在 Redis 中设置一个空值的 Key 来应对下次同样的请求。这样就可以直接返回一个空值，但是不能避免每次不同 Key 值的请求。
  3. 利用布隆过滤器将所有可能存在的请求的值放在该过滤器中，当用户发起请求时，首先校验请求是否存在于布隆过滤器中，如果不存在，拒绝本次请求；否则，在进行正常的查询操作。

布隆过滤器：对给定元素进行哈希计算，根据得到的值将元素存放在对应的位置，但是可能出现不同元素得到相同哈希值的情况，所以过滤器可能出现误判（布隆过滤器说该元素存在，可能不存在，但是，如果说不存在，那么肯定不存在）

缓存击穿

描述：

缓存击穿与缓存穿透不同之处在于，要查询的数据在 Redis 中没有，但是在数据库中存在（一般是缓存到期问题）。这时如果大量的并发请求抵达，导致大量的请求直接落在了数据库上，可能导致数据库宕机。

解决方案：

  1. 增加热点数据的过期时间，甚至设置成永不过期，避免热点数据在大量请求时出现过期的情况。

  2. 增加双重检验锁来限制请求查询数据库的次数。

缓存雪崩

描述：

缓存雪崩是指缓存中大量的数据在同一时间段到期，而导致大量的请求进入数据库进行查询。与缓存击穿不同的是，缓存击穿是指的同一条数据，而缓存雪崩是指的不同的数据。

解决方案：

  1. 缓存数据时尽量避免设置同一时刻过期，将过期时间进行随机设置。
  2. 如果是分布式部署，可以将不同的热点数据分散在不同的服务器下。
  3. 设置热点数据永不过期。
