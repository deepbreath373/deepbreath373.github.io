一致性Hash原理

发表于 2021-10-26 更新于 2022-01-22 分类于 Redis
本文字数： 1.1k

利用一致性 Hash 实现负载均衡

普通Hash出现的问题

为了保证 Redis 的高并发、高可用，我们通常采用搭建 Redis 集群并且采用主从复制、读写分离的方法。当缓存的数量超过一定值的时候，就需要对现有的 Redis 集群进行分库分表的操作。但是，如果采用随机存储的方式，我们存储的数据可能存储在任何一组 Redis 中，当需要查询的时候，不得不遍历所有的 Redis 服务器才能得到想要的数据。因此，可以采用对存储的数据进行 Hash 、取模的方式，来确定存储在哪一组 Redis 服务器中。

上图中，采取 hash(data)%3 的方式定位了 Redis 服务器的存储位置，在下次取值的时候也可以采用同样的计算方法就能快速定位数据所在的 Redis 服务器。

使用 Hash 取模的方法，虽然避免了遍历 Redis 服务器取值的问题，但是，如果 Redis 服务器发生了变动，数量进行了增加或减少，那么所有缓存的位置就会发生变化，即 Hash 取模的计算结果都发生了改动。

为了避免上述情况的发生，就有了一致性 Hash 算法的出现。

一致性Hash算法原理

一致性 Hash 算法也是通过取模的方式来确定数据的存储位置，但是，有别于上述所用的对服务器数量进行取模，一致性 Hash 采用的是对 2 的 32 次方取模。在该算法中，将 Hash 空间比作一个以 0 开始，以 2 的 32 次方减一结束的圆环。

将每个服务器使用服务器的 IP 或者主机名作为关键字进行 Hash ，得到一个 Hash 值作为该服务器在圆环上的位置。

而数据的存储也是用相同的 Hash 函数来计算对应的 Hash 值，借此来确定数据在圆环上的位置，接着从其位置顺时针查找，将查找到的首个服务器作为其存储的服务器。

一致性Hash算法的容错性和可扩展性

假设服务器 B 宕机，那么原本存储在服务器 B 的数据 b 就会被重新定位，定位到其顺时针最近的服务器 C 上，而原本的服务器 A 和服务器 B 不用发生任何改变。

而在环上新增一个服务器，也只会导致部分数据发生重定位。

可以发现，不论是服务器的增加还是减少，所影响的数据只是少部分，因此，该算法拥有不错的容错性和可扩展性。

数据倾斜问题

在服务节点较少的情况下，可能导致节点集中在环的某一侧，而导致的数据倾斜问题，即大部分数据的缓存都落在了某一台服务器上。

大量的数据集中存储在了服务器 A 上，而服务器 B 只有少量的数据存储。为了解决该问题，一致性 Hash 算法引入了虚拟节点机制，即对每个服务器节点进行多次 Hash ，每个位置都放置一个服务节点，称为虚拟节点。

数据的存储方式不变，只需要增加一步虚拟节点到实际服务器节点的映射即可。这样，即使服务器节点较少，也可以通过虚拟节点的方式使得数据分布的均匀。
